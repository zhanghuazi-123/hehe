# 预判注入（Anticipatory Context Injection, ACI）

> 设计理念文档 · 作者：Yuanda + Claude Opus 4.7 · 2026-04-22

---

## 一、问题的根源

现代 LLM Agent 的工作方式是一个严格的串行循环：

```
用户输入
  → LLM 思考（需要什么？）
  → 调用工具 A（等待结果）
  → LLM 思考（还需要什么？）
  → 调用工具 B（等待结果）
  → LLM 思考（够了吗？）
  → 输出回答
```

这带来三个实质性问题：

1. **延迟叠加**：每一步都要等 LLM 完成推理才能开始工具调用，工具调用完才能开始下一次推理，延迟线性累加。
2. **资源空转**：LLM 在等工具结果时闲置，工具在等 LLM 决策时闲置。
3. **记忆盲区**：LLM 每次开口时只看到消息本身，需要自己决定"我还需要哪些信息"——这是一种被动、事后的信息获取方式。

**根本问题**：系统没有利用已经知道的信息——它知道用户的历史习惯，知道当前时间，知道任务模式，却每次都等 LLM 现场决定。

---

## 二、核心理念

> **在模型开口之前，系统主动预判它需要什么，提前查好，直接注入。**

这一句话包含三个关键动作：

- **预判**：不等模型问，系统先推断意图
- **提前执行**：并行执行所需的只读工具/查询
- **直接注入**：结果在 system prompt 里等模型，模型一开眼就看到

模型从"主动调工具的猎人"变成"被喂好信息的厨师"——它只需要处理信息，不需要采集信息。

## 三、三类预判场景

### 场景 A：语义记忆预判（Semantic Memory Prefetch）

**现在**：LLM 看到消息 → 意识到需要记忆 → 调用 `search_memory` → 等结果 → 再思考。

**预判**：消息进来 → 注入器用向量相似度检索相关记忆 → 记忆已经在 prompt 里了。

```
用户说："帮我回顾一下上次我们聊的 TICK 机制"
  ↓ 注入器语义检索：TICK、心跳、时间感知
  ↓ 相关记忆 3 条直接注入
  ↓ 模型开口时已经"想起来了"
```

这是 ACI 中最稳妥、零风险的一类——记忆只是被更好地检索出来，不涉及任何副作用。

---

### 场景 B：工具链模式预判（Pattern-Based Tool Prefetch）

**洞察**：用户对同一类任务，几乎每次都走相同的工具调用序列。这些序列是可以学习和记忆的。

```
历史记录：
  "写今日早报" → fetch(天气) → fetch(热点新闻) → read(日程) → 写文本

下次用户说"写今日早报"：
  ↓ 意图匹配：命中"写早报"模式
  ↓ 并行预执行：fetch(天气) + fetch(热点) + read(日程)  ← 同时执行
  ↓ 三份结果注入 prompt
  ↓ 模型直接写，省掉 3 次工具调用往返
```

工具链的模式来源：
- **人工配置**：用户/开发者明确定义意图 → 工具链映射
- **自动学习**：识别器记录每次任务的工具调用序列，积累后自动归纳模式

---

### 场景 C：定时预热注入（Scheduled Warm Cache）

**洞察**：有些内容和具体对话无关，是周期性、可预期的——天气、新闻、行情、日历。这些完全不需要等用户开口。

```
每天 08:00 (cron)：
  → 查天气 → 存入 prefetch_cache（有效期 1h）
  → 查热点新闻 → 存入 prefetch_cache（有效期 30min）
  → 读日历 → 存入 prefetch_cache（有效期 24h）

当天任何时候 Jarvis 被唤醒：
  ↓ 注入器：今天的天气/新闻/日历已经在缓存里
  ↓ 直接注入
  ↓ 不需要任何工具调用
```

这是 ACI 中确定性最高的一类——不需要意图识别，纯粹靠时间触发，结果有明确时效。

---

## 四、核心约束：什么能预判，什么不能

并不是所有工具都适合预判执行。**只有满足以下条件的工具才能进入预判范围**：

| 条件 | 说明 | 反例 |
|---|---|---|
| **只读（Read-only）** | 执行不会改变任何外部状态 | `write_file`、`send_message` 不能预判 |
| **幂等（Idempotent）** | 多次执行结果相同 | 某些随机 API 不适合 |
| **低副作用** | 没有配额消耗或成本可以接受 | 付费 API 要设置预判频率上限 |
| **结果有时效** | 结果在一定时间内有效，过期应丢弃 | 实时价格需要极短 TTL |

**预判只是"提前执行"，不是"替代决策"**。如果预判错了（工具被调用但模型没用到这个结果），最坏情况是浪费了一次查询，上下文多了一段无用信息——不会造成功能错误。

## 五、置信度机制

预判结果的注入方式取决于置信度：

```
置信度 > 0.85  →  直接注入 system prompt（模型看到即用）
置信度 0.5~0.85 →  轻提示："可能需要以下信息，已备好，按需参考"
置信度 < 0.5   →  不注入（避免污染上下文）
```

置信度来源：
- **模式匹配得分**：消息与历史工具链的语义相似度
- **定时任务**：置信度固定为 1.0（时间触发，确定性最高）
- **规则匹配**：正则/关键词命中，固定 0.9

## 六、注入器的新角色

ACI 实现后，注入器（injector.js）的职责从：

> 检索相关记忆 → 打包上下文

扩展为：

> 检索相关记忆 **+** 预判意图 **+** 并行预执行工具 **+** 消费预热缓存 → 打包完整上下文

整个流程变成：

```
消息到达
  ├─ [并行] 语义向量检索记忆
  ├─ [并行] 意图匹配 → 触发工具链预执行
  └─ [并行] 读取有效的 prefetch_cache

  ↓ 汇总（deadline = 注入器超时，默认 1.5s）
  ↓ 组装 injection 对象
  ↓ LLM 调用，上下文已就绪
```

**关键设计**：注入器有超时上限（1.5s）。如果某个预判工具慢了，结果不等——直接用已有内容。宁可少注入，不能拖慢响应。

## 七、与学界的对照

学界在 2025-2026 年独立提出了相同方向，代表性工作：

| 论文 | 核心贡献 | 与 ACI 的差异 |
|---|---|---|
| [PASTE](https://arxiv.org/abs/2603.18897)（2026.03）| 模式感知投机工具执行，降低 48.5% 延迟 | 在推理框架层（SGLang）实现，非应用层 |
| [B-PASTE](https://arxiv.org/abs/2604.16469)（2026.04）| 多分支投机执行 | 面向推理引擎，不涉及记忆系统 |
| [ContextAgent](https://arxiv.org/abs/2505.14668)（2025.05）| 主动感知上下文，预判工具调用 | 需要额外训练，依赖 fine-tuned 小模型 |
| [Speculative Actions](https://arxiv.org/pdf/2510.04371)（2025.10）| 快模型预测、慢模型验证 | 分布式推理架构，不适合单机 Agent |

**ACI 的差异化定位**：

1. **应用层实现**：在 prompt 组装层做，不需要改动推理引擎，对任何 LLM API 透明
2. **记忆系统深度集成**：预判不只针对工具调用，同时优化记忆检索
3. **定时预热机制**：Scheduled Warm Cache 是独有的，学界没有专门研究
4. **轻量无依赖**：不需要额外训练模型，基于向量检索 + 规则就能跑

## 八、实现路径（分阶段）

### Phase 1：语义记忆检索（基础设施）
- 引入 `sqlite-vec` + 本地 embedding 模型（bge-small-zh）
- 记忆写入时生成 embedding
- `searchRelevantMemories` 改为向量检索 + FTS5 融合排序
- 解决 n-gram 噪音问题

### Phase 2：定时预热缓存（确定性预判）
- 新增 `prefetch_cache` 表（source, content, fetched_at, expires_at, tags）
- 新增 `prefetch/runner.js`，供 cron 和定时任务调用
- 注入器新增 `getValidPrefetchCache()` 步骤

### Phase 3：工具链模式学习（自适应预判）
- 识别器记录每次任务的工具调用序列
- 新增 `tool_patterns` 表存储意图→工具链映射
- 注入器新增意图匹配 + 并行工具预执行逻辑
- 引入置信度门槛控制注入行为

---

## 九、一句话

> **让系统比模型更早知道模型需要什么。**

这不是让模型更聪明，而是让系统更有预见性——把信息采集的延迟从"推理时"提前到"消息到达时"，把串行的工具调用变成并行的准备工作。
